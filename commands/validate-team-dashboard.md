# Validate Team Dashboard

Aggregate validation posture across all team projects into a single shared dashboard. Shows coverage metrics, posture scores, regression trends, and ownership assignments so engineering leads can identify under-validated projects at a glance.

## Usage

```
/validate-team-dashboard                                    # Collect + render full dashboard
/validate-team-dashboard --json                             # Machine-readable snapshot JSON
/validate-team-dashboard --html                             # Also write .vf/team/dashboard.html
/validate-team-dashboard --no-collect                       # Render from cached snapshot only
/validate-team-dashboard --register /path/to/project        # Register a project in the registry
/validate-team-dashboard --assign 'project/journey' owner   # Assign a journey to an owner
/validate-team-dashboard --team-dir /shared/.vf/team        # Override team registry path
```

### Parameters

- **--json** — Output raw `snapshot.json` to stdout (skips terminal table). For CI pipelines.
- **--html** — Write `.vf/team/dashboard.html` in addition to Markdown.
- **--no-collect** — Skip `collect-team-metrics.sh`; use cached snapshot. Useful for re-rendering.
- **--register** — Add a project path to `.vf/team/projects.json` before collecting.
- **--assign** — Set ownership for a specific journey (format: `project/journey owner`).
- **--team-dir** — Override default `.vf/team` registry directory for shared team locations.

## Architecture

```
User: "/validate-team-dashboard"
              |
              v
    [team-validation-dashboard skill]
              |
              +-- scripts/collect-team-metrics.sh
              |       reads  .vf/team/projects.json
              |       scans  {project}/.vf/benchmarks/
              |       writes .vf/team/snapshot.json
              |
              +-- scripts/team-dashboard.sh
              |       reads  .vf/team/snapshot.json
              |       reads  .vf/team/ownership.json
              |       renders terminal table (ANSI color)
              |       writes .vf/team/dashboard.md
              |
              +-- [Identify projects needing attention]
              |       critical  < 60 → Run /validate immediately
              |       warning  60–79 → Review + assign owners
              |       healthy   ≥ 80 → No action required
              |
              +-- [Optional: assign-ownership.sh]
                      reads/writes .vf/team/ownership.json
```

## Dashboard Columns

| Column | Source | Meaning |
|--------|--------|---------|
| Project | `projects.json` | Registered project name |
| Last Validated | `.vf/benchmarks/` | ISO date of most recent benchmark |
| Score | `weighted_score` | 0–100 posture score |
| Grade | Derived | A/B/C/D/F |
| Coverage | `coverage_pct` | % of journeys with evidence |
| Regressions | `regressions` | Count of regressions found |
| Owners | `ownership.json` | Assigned team members |

Score thresholds: **Green** (≥80), **Yellow** (60–79), **Red** (<60).

## Data Availability Note

The `posture_score` field is populated from the `aggregate.weighted_score` in existing
`/validate-benchmark` output. The `coverage_pct`, `regression_count`, and `journey_count`
fields require an enriched benchmark format (not yet generated by the current toolset)
and will display as 0 for projects validated with the existing tools.

## Registering Projects

Before collecting metrics, register each project in `.vf/team/projects.json`:

```
/validate-team-dashboard --register /path/to/my-app
```

Or edit `.vf/team/projects.json` directly:

```json
{
  "team": "engineering",
  "projects": [
    { "name": "my-app",      "path": "/path/to/my-app" },
    { "name": "api-service", "path": "/path/to/api-service" }
  ]
}
```

## Assigning Ownership

Assign a team member to a specific journey:

```
/validate-team-dashboard --assign 'my-app/login-flow' alice
```

Or via script directly:

```bash
bash scripts/assign-ownership.sh --project my-app --journey login-flow --owner alice
bash scripts/assign-ownership.sh --list                    # All assignments
bash scripts/assign-ownership.sh --list-project my-app     # One project
bash scripts/assign-ownership.sh --remove --project my-app --journey login-flow
```

Ownership is stored in `.vf/team/ownership.json` and rendered in the dashboard table.

## Priority Report

After rendering, the skill classifies projects by urgency:

```markdown
## Projects Needing Attention

### Critical (< 60)
- **project-alpha** — Score: 42 (F). Last validated: 2026-03-01. 3 regressions.
  → Run /validate immediately. Assign owners to failing journeys.

### Needs Attention (60–79)
- **project-beta** — Score: 68 (D). Last validated: 2026-03-28. 1 regression.
  → Review failing journeys. Assign owners. Schedule validation.

### Healthy (≥ 80)
- **project-gamma** — Score: 91 (A). Last validated: 2026-04-09.
  → No action required. Monitor trend at next benchmark cycle.
```
