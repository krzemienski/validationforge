ValidationForge ships **15 slash commands** across two families. The
**Validation** family is the user-facing interface (nine `/validate*`
commands plus `/vf-setup`). The **Forge** family is the orchestration
layer (six `/forge-*` commands) typically invoked by skills or agents
with explicit tool restrictions. Both families access the same 7-phase
pipeline.

<Aside type="tip" title="Jump to a command">
  Every command below links to its own page with the full usage, flags,
  and behavior details ported from the source `commands/*.md` files.
</Aside>

## Validation Commands (9)

The user-facing interface. No `allowed-tools` restriction — these use
whatever Claude Code makes available in the session.

| # | Command                                           | Description                                                                                                  |
| - | ------------------------------------------------- | ------------------------------------------------------------------------------------------------------------ |
| 1 | [`/validate`](/commands/validate/)                 | Run full end-to-end validation — detect platform, map journeys, capture evidence, write verdicts.            |
| 2 | [`/validate-plan`](/commands/validate-plan/)       | Analyze codebase and generate a validation plan with PASS criteria — no execution.                           |
| 3 | [`/validate-audit`](/commands/validate-audit/)     | Read-only validation audit — captures evidence and classifies findings without modifying code.               |
| 4 | [`/validate-fix`](/commands/validate-fix/)         | Fix validation failures and re-validate until all journeys pass (3-strike limit).                            |
| 5 | [`/validate-ci`](/commands/validate-ci/)           | Non-interactive CI/CD mode — auto-execute full validation pipeline with exit codes.                          |
| 6 | [`/validate-team`](/commands/validate-team/)       | Spawn coordinated validation agents across platforms with evidence handoff.                                  |
| 7 | [`/validate-sweep`](/commands/validate-sweep/)     | Autonomous validation loop — detect, validate, fix, re-validate until PASS or max attempts.                  |
| 8 | [`/validate-benchmark`](/commands/validate-benchmark/) | Benchmark validation coverage, speed, and evidence quality against baseline metrics.                      |
| 9 | [`/vf-setup`](/commands/vf-setup/)                 | Setup and configure ValidationForge for a project or globally.                                               |

## Validation vs. Forge Commands

Both families access the same pipeline. The difference:

- **Validation commands** (`/validate*`, `/vf-setup`) are the user-facing
  interface. No `allowed-tools` restriction — they use whatever Claude
  Code makes available in the session.
- **Forge commands** (`/forge-*`) are the orchestration layer. Each
  specifies `allowed-tools` in frontmatter to constrain which Claude
  Code tools the command may invoke. Forge commands are typically
  called by skills or agents rather than directly by users.

## The 7-Phase Pipeline

Every validation command drives some subset of the same seven phases:

1. **RESEARCH** — standards, best practices, and applicable criteria.
2. **PLAN** — journeys, per-journey PASS criteria, and evidence
   requirements.
3. **PREFLIGHT** — build compiles, services running, MCP servers
   available. If preflight fails, the pipeline stops.
4. **EXECUTE** — run each journey against the real system and capture
   evidence into `e2e-evidence/<journey-slug>/`.
5. **ANALYZE** — root-cause investigation for FAILs using sequential
   thinking.
6. **VERDICT** — evidence-backed PASS / FAIL per journey, consolidated
   into a unified `e2e-evidence/report.md`.
7. **SHIP** — production readiness audit and a deploy decision.

See the [Pipeline Matrix](#command-pipeline-matrix) above for the exact
phases each command runs.
