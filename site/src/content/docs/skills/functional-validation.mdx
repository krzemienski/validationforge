# Functional Validation Protocol

This is the **core validation protocol** in ValidationForge — the one every
platform validator ultimately defers to. It defines how to go from a built
system to an evidence-backed PASS/FAIL verdict without ever writing a mock.

## When to Use

Use `functional-validation` whenever you need to prove a feature works end-to-end:

- A journey planner (`create-validation-plan`) has produced PASS criteria and
  you need to exercise them against the real system.
- A platform-specific skill ([`web-validation`](/skills/web-validation/),
  [`api-validation`](/skills/api-validation/),
  [`ios-validation`](/skills/ios-validation/), `cli-validation`,
  `fullstack-validation`) needs the shared "capture evidence, write verdict"
  spine.
- You are about to claim a task is done and need a repeatable procedure
  for proving it works.

Does **not** handle mock blocking (use
[`no-mocking-validation-gates`](/skills/no-mocking-validation-gates/)) or
evidence examination at completion (use
[`gate-validation-discipline`](/skills/gate-validation-discipline/)).

## The Iron Rule

```
NEVER create mocks, stubs, test doubles, or test files.
NEVER import jest, vitest, pytest, XCTest, or any test framework.
ALWAYS validate through the same interfaces real users experience.
ALWAYS capture evidence that proves the feature works end-to-end.
```

## Before You Validate

1. **Is the real system running?** (not a dev server with mocks)
2. **Can I access it as a user would?** (browser, simulator, CLI — not a test harness)
3. **Are all dependencies real?** (real DB, real API keys, real network)
4. **Do I have PASS criteria written down?** (specific, observable, measurable)
5. **Am I capturing evidence to files?** (not just observing in terminal)

## Platform Detection

Detect the platform FIRST. Wrong platform = wrong validation approach.

| Priority | Platform | Indicators | Approach | Reference |
|----------|----------|-----------|----------|-----------|
| 1 | iOS/macOS | `*.xcodeproj`, `Package.swift` | Xcode build, simulator, idb/simctl | [`ios-validation`](/skills/ios-validation/) |
| 2 | Web | `package.json` + React/Vue/Next | Dev server, browser, Playwright MCP | [`web-validation`](/skills/web-validation/) |
| 3 | CLI | `main.go`, `Cargo.toml`, `cli.py` | Build binary, execute, capture stdout | [`cli-validation`](https://github.com/krzemienski/validationforge/blob/main/skills/cli-validation/SKILL.md) |
| 4 | API | `server.ts`, `app.py` + routes | Start server, curl endpoints | [`api-validation`](/skills/api-validation/) |
| 5 | Full-Stack | Frontend + Backend present | Bottom-up: DB → API → Frontend | [`fullstack-validation`](https://github.com/krzemienski/validationforge/blob/main/skills/fullstack-validation/SKILL.md) |

## The 4-Step Protocol

### Step 1: Build & Launch

Build the real system with all real dependencies. If the build fails, that IS
your first finding — capture the full build output as evidence and stop.

### Step 2: Exercise Through UI

Interact as a real user would — browser, simulator, CLI binary, or curl. No
REPL imports, no direct function calls, no "unit test that approximates the UI."

### Step 3: Capture Evidence

Save to `e2e-evidence/` — screenshots, response bodies, CLI output, logs.
Evidence must be READ and DESCRIBED, not just confirmed to exist. A 0-byte
file is not evidence.

### Step 4: Write Verdict

For each criterion, cite specific evidence with file paths and quoted content.
No citation = no PASS.

## Multi-Platform Validation Order

For full-stack apps, validate bottom-up. Each layer must PASS before testing
above it.

```
Database/Infra (validate FIRST)
        ↓
Business Logic
        ↓
API Endpoints
        ↓
Frontend UI (validate LAST)
```

**Why:** If the DB is broken, every API test fails with misleading errors, and
every frontend test fails with a completely different misleading error. Start
at the data layer to find causes, not symptoms.

## Verdict Format

Every verdict — per criterion — looks like this:

```markdown
### Criterion: [What was required]
**PASS** / **FAIL**
Evidence: `e2e-evidence/[file]` — [What I actually saw, quoted specifically]
```

Without cited evidence, a verdict is not a verdict — it's a guess. See
[`gate-validation-discipline`](/skills/gate-validation-discipline/) for the
evidence-examination rules that enforce this.

## Security Policy

This skill executes real systems for validation. It never introduces new
functionality, never disables security checks, and never bypasses auth — it
validates through the same security boundaries real users encounter.
