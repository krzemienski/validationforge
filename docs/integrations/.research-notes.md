# Ecosystem Integration — Research Notes

> **Purpose:** Source of truth for the three VF ecosystem integration guides (OMC, ECC, Superpowers).
> **Audience:** Contributors drafting `vf-with-omc.md`, `vf-with-ecc.md`, `vf-with-superpowers.md`.
> **Status:** Working document — update as guides are drafted and new facts surface.
>
> This file is intentionally kept inside `docs/integrations/` (alongside the guides) rather than in `plans/`
> so it travels with the guide source. It is not published — the leading `.` keeps it out of docs indexes,
> and tooling treats it as a local working note.

---

## Scope

Each of the three integration guides must answer the same three questions:

1. **What does the upstream plugin do that VF does NOT do?** (why pair them at all)
2. **Where does the handoff from upstream → VF live?** (which command, which artifact, which hook)
3. **What concrete conflicts exist?** (hook ordering, test-file gates, rules precedence, evidence overlap)

The notes below capture the raw research that feeds those three answers for each plugin.

## 1. oh-my-claudecode (OMC)

### Sources
- `plans/reports/competitive-analysis-260308.md` — VF-authored competitive analysis (primary).
- `docs/competitive-analysis.md` — Published competitive analysis (matches 260308 report).
- OMC installed plugin cache at `~/.claude/plugins/cache/` (referenced in source reports).
- Upstream: https://github.com/oh-my-claudecode

### Summary

OMC is a **multi-agent orchestration platform**. It ships execution loops (ralph, autopilot, ultrawork,
team), a consensus planning system (ralplan with Planner/Architect/Critic deliberation), and a bundled
MCP server. Its mental model is "coordinate N specialized agents to produce working software."

Validation in OMC is incidental — it has no evidence system, no no-mock stance, no platform-specific
validation, and no test-file gates.

### Component inventory (from competitive-analysis-260308.md)

| Category | OMC v4.7.7 | VF |
|----------|-----------:|---:|
| Skills   | 37–38      | 40 |
| Agents   | 25–28      | 5  |
| Commands | 0–38       | 15 |
| Rules    | 0 (CLAUDE.md-injected) | 8 |
| Hooks    | ~96 entries / 14 unique | 7 |
| MCP servers | 1 (bundled) | 0 |
| Templates | 2 | 4 |

### Key OMC capabilities

- **Execution modes:** ralph (self-referential loops), autopilot (idea → code), ultrawork (max parallelism), team (coordinated agents through plan → prd → exec → verify → fix stages).
- **Consensus planning:** ralplan loop with Planner, Architect, Critic; RALPLAN-DR deliberation mode.
- **Agent catalog:** analyst, architect, debugger, executor, verifier, critic, designer, planner, writer, tdd-guide, test-engineer, etc.
- **State management:** MCP-backed state read/write/clear with session scoping; notepad; project memory.
- **Bundled MCP server:** LSP, AST grep, Python REPL, notepad, project memory, state management.
- **Model routing:** haiku / sonnet / opus per task for cost optimization.
- **Notifications:** Discord, Telegram, Slack integration.
- **Setup:** `omc-setup` is mature — idempotent, supports local + global install, configures MCP.

### VF ↔ OMC overlap / conflict points

- **No direct rule conflict.** OMC injects into CLAUDE.md; VF ships standalone rules under `rules/`.
- **Test-file gates:** OMC's tdd-guide / test-engineer agents CAN produce test files. VF's
  `block-test-files` hook will block these if both plugins are active and OMC attempts to write into
  the project root. Expected guide advice: scope OMC test output to a directory VF ignores, or treat
  OMC's TDD mode as "off" during the validation phase.
- **Evidence directory:** OMC has no concept of `e2e-evidence/`. No overlap.
- **Execution loops:** OMC's ralph and VF's `validate-sweep` are both autonomous loops; the guide
  needs to make clear that ralph builds, validate-sweep validates-and-fixes, with a 3-attempt cap.

### Recommended pairing
`OMC builds / orchestrates → VF validates the shipped artifact`. The handoff point is
"OMC reports done" → run `/validate` or `/validate-sweep`.

## 2. everything-claude-code (ECC)

### Sources
- `plans/reports/competitive-analysis-260308.md`
- `docs/competitive-analysis.md`
- Upstream: https://github.com/everything-claude-code

### Summary

ECC is a **broad development toolkit with language-specific rule sets**. It covers Django, SpringBoot,
Go, Swift, C++, Java, Python, and frontend patterns, plus security review, TDD, and article-writing
skills. Its `configure-ecc` / `install.sh` installs rules into `.claude/` for either Claude Code or
Cursor based on the languages the user selects.

ECC's testing stance is traditional — mocks, stubs, unit tests, 80% coverage mandate. This is the
**direct philosophical opposite** of VF's no-mock stance and is the single most important conflict to
call out in the integration guide.

### Component inventory

| Category | ECC v1.7.0 | VF |
|----------|-----------:|---:|
| Skills   | 65 / ~15 (depends on report) | 40 |
| Agents   | 13–17      | 5  |
| Rules    | 29 (9 common + per-language) | 8 |
| Hooks    | ~13–16 unique | 7 |
| Commands | ~37 unique | 15 |
| Install  | `install.sh --target claude\|cursor <languages>` | `/vf-setup` |

### Key ECC capabilities

- **Language specialization:** golang-patterns, swift-concurrency, django-tdd, springboot-patterns, python-typing, etc.
- **TDD workflow:** red/green/improve loop with 80% coverage mandate.
- **Security-first:** secret detection, input validation, injection prevention hooks.
- **E2E testing:** `e2e-runner` agent using Playwright.
- **Content skills:** article-writing, investor-materials, market-research (tangential to VF's mission).
- **Setup:** `configure-ecc` / `install.sh` with per-language rule selection.
- **Proactive agent triggers:** agents self-activate based on task type.

### VF ↔ ECC overlap / conflict points

- **Test files.** ECC actively *encourages* creating test files. VF actively *blocks* them with the
  `block-test-files` PreToolUse hook. Guide needs a clear policy:
    - Option A: run ECC's TDD phase *before* VF hooks are installed (install VF last).
    - Option B: scope VF's block-test-files hook to exclude the `tests/` or `__tests__/` directory where ECC writes.
    - Option C: stage ECC's TDD work in an isolated worktree, merge into main once VF evidence proves the real system works.
- **Rules:** ECC installs 29 rules into `.claude/`, VF installs 8. Names do not collide based on the
  current audit; but users running both should inspect `.claude/rules/` for accidental overwrites.
- **Hooks:** ECC has ~13 unique hooks, VF has 7. No direct event collision expected, but
  PostToolUse ordering on `Edit`/`Write` is worth testing — VF's `mock-detection` and
  `evidence-quality-check` should fire *after* ECC's own file-op hooks for the warnings to be useful.
- **E2E testing:** ECC's `e2e-runner` uses Playwright; VF's `playwright-validation` uses Playwright.
  Both can coexist — ECC's version is generic, VF's produces evidence for `e2e-evidence/`.

### Recommended pairing
`ECC enforces language rules and security → VF validates the running app with evidence`.
The handoff point is "ECC's static/unit gates green" → run `/validate`.

## 3. Superpowers

### Sources
- https://github.com/obra/superpowers (repo, commands/, skills/ directories)
- https://github.com/obra/superpowers/tree/main/commands
- https://github.com/obra/superpowers/tree/main/skills
- https://blog.fsck.com/2025/10/09/superpowers/ (author's announcement blog post)
- Install: `/plugin marketplace add obra/superpowers-marketplace` then `/plugin install superpowers@superpowers-marketplace` (requires Claude Code v2.0.13+)

### Summary

Superpowers is a **plan-and-execute TDD methodology plugin**. Its core loop is
`brainstorm → write plan → execute plan`, where the execution phase dispatches fresh subagents per
task and enforces a RED/GREEN TDD discipline with two-stage code review (spec compliance, then
code quality). It is intentionally small, opinionated, and built around "skills as prompts."

Superpowers' author (Jesse Vincent / obra) frames skills as "the interesting part" — Markdown
`SKILL.md` files that teach Claude reusable procedures, with the property that Claude MUST use a
relevant skill when one exists. A session hook loads `getting-started/SKILL.md` on startup so the
skill system is live from the first turn.

### Slash commands (confirmed)

From `commands/` directory listing:

- **`/brainstorm`** — Refines a rough idea through questions, explores alternatives, presents the
  design in sections for validation before a plan is written.
- **`/write-plan`** — Divides the work into "bite-sized tasks (2–5 minutes each)" with exact file
  paths, complete code, and verification steps.
- **`/execute-plan`** — Either dispatches fresh subagents per task with a two-stage review, or
  executes in batches with human checkpoints.

These are the three primary user-facing entry points. Installation also adds a session startup
hook that loads the `using-superpowers` skill.

### TDD methodology (RED-GREEN-REFACTOR via subagents)

Superpowers' `test-driven-development` skill enforces a strict sequence:

> write failing test → watch it fail (RED) →
> write minimal code → watch it pass (GREEN) →
> commit → refactor.

Additional properties captured from the sources:

- Code written **before** a test is **deleted** — the skill explicitly says so.
- The same TDD discipline is applied to the skills themselves: Claude "pressure-tests" a new skill
  with subagents running realistic scenarios before the skill is considered ready.
- Execution is **subagent-driven** — each task in the plan is handed to a fresh subagent so the
  context is clean, and the parent agent reviews the subagent's output in two stages:
  (1) spec compliance, (2) code quality.
- `dispatching-parallel-agents` is a first-class skill — Superpowers can run multiple TDD loops in
  parallel against isolated git worktrees.

### Skill catalog (confirmed directory list)

From `skills/` directory listing:

| Category | Skills |
|----------|--------|
| Testing | `test-driven-development` |
| Debugging | `systematic-debugging`, `verification-before-completion` |
| Collaboration | `brainstorming`, `writing-plans`, `executing-plans`, `dispatching-parallel-agents`, `requesting-code-review`, `receiving-code-review`, `using-git-worktrees`, `finishing-a-development-branch`, `subagent-driven-development` |
| Meta | `writing-skills`, `using-superpowers` |

Key properties of Superpowers skills:

- Each skill is a single `SKILL.md` Markdown file under `skills/<skill-name>/SKILL.md`.
- Skill usage is **mandatory** — if a relevant skill exists, Claude is required to use it.
- Skills are **self-improving** — Claude can author new skills to capture learned procedures, and
  the `writing-skills` skill defines that meta-process.
- Skills are **discovered via a script** that searches the skills directory at session start.

### VF ↔ Superpowers overlap / conflict points

- **Test files.** Superpowers writes test files as part of its RED phase. VF's `block-test-files`
  hook will block these if both plugins are active and Superpowers writes into a path the hook
  guards. The guide must recommend one of:
    - Run Superpowers' RED/GREEN loop in a dedicated worktree (Superpowers already uses git worktrees
      as a first-class primitive — this maps cleanly).
    - Configure VF's block-test-files hook to exempt the path where Superpowers puts tests
      (typically `test/`, `tests/`, or `__tests__/`).
    - Accept that Superpowers' RED phase completes before VF's hooks are loaded (install VF last).
- **Verification skill overlap.** Superpowers ships `verification-before-completion`;
  VF ships `verification-before-completion` (same name, different content). Both skills are opinions
  about when to declare "done." Guide needs to decide which takes precedence per phase:
  Superpowers during the TDD cycle, VF during the validation phase.
- **Mandatory-skill semantics.** Superpowers' "must use a relevant skill" rule is a strong claim.
  The guide should note that VF's no-mock rule is the stronger rule during the validation phase
  (Iron Rules 1, 2, 8 in `CLAUDE.md`) and overrides any Superpowers skill that would produce mocks.
- **Evidence directory.** Superpowers has no concept of `e2e-evidence/`. No overlap.
- **Worktrees.** Superpowers uses worktrees heavily; VF's auto-claude mode also runs in worktrees.
  This is complementary — the guide can recommend a pattern where Superpowers' TDD loop runs in
  its own worktree, and VF validates the merged output on main.

### Recommended pairing
`Superpowers drives the brainstorm → write-plan → execute-plan TDD loop → VF validates the
assembled system end-to-end so we know the passing unit tests weren't just passing against mocks`.
The handoff point is "Superpowers' `/execute-plan` reports GREEN across all tasks" → run `/validate`.

## Cross-cutting notes (all three guides)

- **Iron Rules take over during the validation phase.** Regardless of which upstream plugin
  produced the code, once VF is invoked the eight Iron Rules from `CLAUDE.md` become binding.
  The single most important one for these guides is Rule 2: "Never create mocks, stubs, test
  doubles, or test files" — this is what causes the real conflicts with ECC and Superpowers.
- **Hook ordering matters.** VF's PreToolUse hooks (`block-test-files`) fire before any file is
  written. If ECC or Superpowers install competing PreToolUse hooks on the same events, the *last
  installed* wins in some Claude Code configurations. Guide should recommend install order:
  upstream plugin first, VF last, so VF's gates are authoritative.
- **Evidence is the contract.** For all three pairings, the final deliverable is a
  `e2e-evidence/report.md` with cited evidence. Upstream plugins own the "build" artifact; VF owns
  the "proof it works" artifact.
- **Diagram convention.** Each guide's combined-workflow Mermaid diagram should use the same
  three swim-lanes: `User` → `Upstream plugin (OMC | ECC | Superpowers)` → `VF`. Consistency
  across the three guides helps readers who compare them.
