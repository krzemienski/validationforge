Verified: Step 1 readiness loop (line 39)
-----------------------------------------
Command form:
  curl -s http://localhost:3000 -o /dev/null -w "%{http_code}" 2>/dev/null | grep -q "200"

Tested with no server running:
  curl output: "000"
  grep -q "200" result: no match (exit 1)
  Loop behavior: 30 iterations × 1s = 30s total wait — correct.

Tested response code parsing: OK.

Note: Loop counts only "200" as ready; any 3xx redirect (e.g., /→/en for i18n
frameworks, /→/login for auth gated SPAs) will NOT match and the wait will
time out despite the server being ready. A more robust test would be
`grep -E "^(2|3)[0-9][0-9]$"` or `test "$CODE" \!= "000"`.

Verified: Step 8 route-check loop
----------------------------------
Command form:
  STATUS=$(curl -s -o /dev/null -w "%{http_code}" "http://localhost:PORT$route")

Tested with no server: STATUS=000. Skill says "Any non-200 response (except
expected redirects) is a finding." — but the bash loop has no whitelist for
redirects; all 301/302 would fire as a finding unless a validator manually
filters. Flag as MEDIUM.

Verified: `lsof -ti:PORT | xargs kill -9` (Common Failures row 1)
----------------------------------------------------------------
lsof: available on macOS. The PIPE form is safe if PORT is numeric but will
kill EVERYTHING listening on PORT without confirmation — potentially harmful
on a workstation running multiple dev servers on the same port.
